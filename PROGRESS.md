# STL Generator — журнал прогресса

## Решение: CSG / булево объединение моделей — не делаем

По плану была задача «булево объединение моделей (CSG)». По согласованию с владельцем снята: надёжный
булев движок мешей (BSP-деревья, компланарные грани, численная устойчивость) в самодостаточном файле без
зависимостей — очень крупная и рискованная по герметичности работа, а практическая польза мала — слайсеры
штатно объединяют пересекающиеся оболочки при нарезке (все видимые модели уже выгружаются в один STL,
`mergedExportTris`). Ограничение переформулировано в README как осознанное. На этом список задач закрыт.

## Этот проход: осевая плотность скругления отдельным контролом (`filletAxisRes`)

`filletSeg` задаёт гладкость ДУГИ (сегменты скругления), а плотность ОСЕВОЙ сетки (плоские участки
между скруглёнными рёбрами) у asym-билдеров раньше жёстко выводилась из него (`max(8, filletSeg*2)`).
Теперь это отдельный параметр «Скругление: плотность осевой сетки (0=авто)» (`filletAxisRes`).

- Хелпер `asymAxisRes(p)`: `0` → прежняя связка `max(8, filletSeg*2)` (обратная совместимость), любое
  `>0` задаёт плотность напрямую (мин. 2), независимо от `filletSeg`.
- Применён во всех 4 asym-путях: скруглённый полый, скруглённый борт, сплошной asym и asym+выгиб.
- Squircle-пути не трогали — там своя механика плотности.

Проверка: `test_rounded_fillet.js` расширен (20/20): auto=0 водонепроницаемо; высокая плотность добавляет
треугольники (при неизменном `filletSeg`), низкая — грубее auto; тот же контрол на скруглённом полом —
водонепроницаемо. Браузер: контрол рендерится (input `data-key="filletAxisRes"`), ошибок нет. Батарея
**476/476 в 17 файлах**.

## Этот проход: Web Worker для проверки герметичности (неблокирующая диагностика)

Проверка водонепроницаемости (`manifoldCheck`) — последний тяжёлый синхронный шаг на главном потоке
(~0.7 с на 120k тр.); прежний `setTimeout(…,0)` только отодвигал её за кадр, но, запустившись, она всё
равно блокировала UI (тот самый «фриз»). Вынесли её в **Web Worker**.

- **Источник воркера = сама `manifoldCheck`, сериализованная через `.toString()`** + крошечный
  onmessage-хендлер. У функции НОЛЬ внешних зависимостей (только Map/Int32Array/Math), поэтому дублирования
  логики нет и сборка не нужна — off-thread крутится ровно та же процедура. Треугольники передаются
  transferable-буфером **Float64Array** (9 double/тр.) — именно Float64, чтобы координаты были бит-в-бит
  как на главном потоке и результат склейки/рёбер совпадал с inline-проверкой.
- Воркер создаётся лениво из Blob-URL; если `Worker` недоступен или заблокирован — fallback на прежний
  `setTimeout`-путь. Токен `statCheckGen` отбрасывает устаревший ответ, если пришла новая пересборка.
- Потолок проверки поднят со 120k до 600k треугольников (раз она больше не блокирует поток).
- **Генерацию геометрии в воркер НЕ выносили — осознанно.** Вся геометрия живёт на верхнем уровне одного
  файла и завязана на глобалы (`paramState`, `logos`), а Node-тесты (470 шт.) вызывают `buildTrisForShape`
  и др. напрямую из top-level scope. Воркер потребовал бы либо дублирования ~2500 строк, либо обёртки всей
  геометрии в функцию — что сломало бы либо принцип «один файл без сборки», либо тест-харнесс. Выигрыш
  мал (генерация уже за debounce + кнопкой «Подтвердить»), риск для рабочего инструмента велик. Поэтому
  вынесли именно диагностику — конкретный задокументированный источник фриза.

Проверка: реальный браузер (Chromium/Playwright) — воркер создаётся, его результат бит-в-бит совпадает с
inline `manifoldCheck` и на герметичном (супереллипсоид 2208 тр.), и на заведомо открытом меше (4 кромки);
бейдж «замкнутый ✓» обновляется через ответ воркера после реальной пересборки; ошибок страницы нет.
Node-батарея без изменений — **470/470** (воркер браузерный, тесты его не трогают).

## Решение: сетчатое дно остаётся только на плоском полом (не борт / не скруглённый)

По плану была задача «сетчатое дно для борта и скруглённого полого». При разборе оказалось, что фича
как сформулирована не имеет осмысленной реализации: диагональная решётка — это СКВОЗНЫЕ отверстия для
дренажа, а вести им некуда. У борта/лотка под карманом сплошная база; у скруглённого полого дно —
купол (ровные ромбы по кривизне определены плохо и практической пользы не несут). По согласованию с
владельцем задача снята; ограничение переформулировано в README как осознанное, а не «на будущее».

## Этот проход: зональная сетка логотипов на ПЛОСКОМ полом контейнере / борте

Раньше зональная сетка (грубая оболочка + мелкая детализация только под логотипом) была у сплошного
куба и у СКРУГЛЁННОГО полого/борта, а у ПЛОСКОГО полого/борта — нет: `logoResolutionHollow` гнал
плотность ВСЕЙ оболочки, поэтому крупная детализация логотипа требовала перегенерации всего меша.
Теперь плоский полый/борт тоже зональный.

- **`buildHollowBox` / `buildRimBox`** приняли `zonesByAxis` + `logoRes`. Когда есть логотипы (и не
  сетчатое дно): база оболочки строится ГРУБОЙ (`baseRes=min(res,4)` — плоская/наклонная стенка
  планарна, нескольких ячеек хватает), а под каждым логотипом добавляются мелкие подразбиения
  (`addLogoZonesToAxis`, как у скруглённых билдеров). Без логотипов — прежняя равномерная сетка
  (plain / только-выгиб / сетчатое дно не меняются).
- **Выгиб стенок**: полноразмерные зоны выгиба (`computeWallBulgeZones`) домешиваются к зонам
  логотипов (`mergeZones`), поэтому купол по-прежнему получает сетку на всю грань.
- **Фикс верхнего обода (нашёл фаззер-стиль баг в тестах).** Зональные точки попадают и в УГЛОВЫЕ
  полосы верхнего обода (`[-hd,-(hd-t)]` и т.п.), а обод раньше собирался одиночными квадратами
  (`addQuad`) + полосами с 2-точечными краями — лишние точки стенки давали T-стык (пример: полый+выгиб+
  лого — 12 открытых кромок; экстремальная толщина стенки + лого дна — 56). Обод переписан как
  **банд-сетка 3×3** (`xLo/xMid/xHi` × `zLo/zMid/zHi`, минус центральный проём) по фактическим срезам
  позиционных массивов — каждая вершина обода теперь совпадает с верхними кромками стенок. То же в борте.
- Диспетчеры плоского полого/борта считают `zones = computeLogoZonesByAxis(taperFn)` (+bulge при выгибе)
  и передают их вместе с `logoResolutionHollow` как плотность зон.

Проверка: `test_hollow_inner_logo.js` расширен (18/18): зональная оболочка с логотипом дешевле
равномерной при той же высокой детализации; логотип у самого вертикального ребра (зона заходит в
угловую полосу обода) — водонепроницаемо; выгиб+лого — водонепроницаемо; борт с логотипом — зонально и
водонепроницаемо. Батарея **470/470 в 17 файлах**.

## Этот проход: раздельное скругление верха и низа супереллипсоида

Единый параметр «Squircle: скругление верха/низа» (`squircleV`) разбит на ДВА независимых —
«скругление верха» (`squircleVTop`) и «скругление низа» (`squircleVBot`). Теперь верх и низ
скругляются по отдельности: 0 на стороне = плоская крышка там; можно сделать куполообразный верх
при плоском дне и наоборот.

- **`buildSuperellipsoid`** (no-logo путь) принимает два экспонента полюсов: `phi>=0` берёт `eVTop`,
  `phi<0` — `eVBot`. При `eV=0` на стороне `seС/seS` дают плоскую крышку (лишние вырожденные кольца
  отфильтровываются), при этом прямая стенка от экватора до крышки строится штатным квадом.
- **`buildSuperellipsoidLogos`** (cube-sphere путь с логотипами): экспонент выбирается по знаку `dy`
  в `seImplicitSurface`/`seImplicitNormal`. Ключ к герметичности — на экваторе (`y=0`) горизонтальное
  сечение решается в `(|x|/hw)^aH+(|z|/hd)^aH=1` НЕЗАВИСИМО от вертикального экспонента, а `ny=0`
  (aV>1), поэтому и поверхность, и нормализованная нормаль непрерывны через `dy=0` даже при разных
  экспонентах верх/низ. Значит бок-грани cube-sphere, пересекающие экватор, остаются водонепроницаемыми.
- **Полый контейнер / борт**: используют `squircleVBot` для скругления ДНА (верх открыт плоским бортом,
  `squircleVTop` — только для сплошного куба). `buildSquircleHollow` не менялся, поменялся лишь источник
  `eV` в диспетчере.
- Диспетчер сплошного куба: `eVTop>0 || eVBot>0` → супереллипсоид с раздельными экспонентами; оба 0 →
  плоский squircle-призм (`buildSquirclePrism`).

Проверка: `test_squircle.js` расширен (миграция `squircleV`→`squircleVTop/Bot`; для полого/борта —
`squircleVBot`) + новая секция «независимое верх/низ» (top-only / bottom-only водонепроницаемы и дают
разный объём, чем оба; асимметрия + логотипы на обоих полюсах и боку). Батарея **464/464 в 17 файлах**.

## Этот проход: несколько моделей (сборка)

«Можно ли добавлять и настраивать больше одной модели?» — да. Дизайн: ГЛОБАЛЫ приложения
(paramState.box, logos, transformState, новый positionState, cachedRawTris/currentTris) всегда держат
состояние АКТИВНОЙ модели — весь существующий код работает без изменений. Каждая модель — запись
{params, logos, rx/ry/rz, px/py/pz, visible, color, rawTris, bbox}; переключение = сохранить-глобалы-в-
запись / загрузить-запись-в-глобалы (`saveActiveModel`/`loadModel`) + обновить панели
(`refreshActiveModelUI`, карточки лого/моделей). Геометрия при переключении НЕ пересобирается — у записи
есть кэш rawTris, активной модели хватает rotate-only пути.

- **Карточки моделей** (как карточки логотипов): добавить (до 12) / дублировать / удалить / скрыть /
  переименовать; активная подсвечена. Новая модель ставится справа от сборки (авто-px). Дубль — рядом с
  оригиналом, логотипы копируются (heightmap разделяется по ссылке — он неизменяемый).
- **Позиция модели (px/py/pz)** — новый живой путь без пересборки (как поворот): двигается только
  THREE-меш (`applyPositionOnly`), позиция применяется к треугольникам лишь при экспорте.
- **Сцена**: THREE.Mesh на модель (свой материал → свой цвет), `mesh` всегда указывает на активный —
  цвет/шейдинг/каркас работают как раньше (шейдинг применяется ко всем). Сетка-пол — по минимальному Y
  всех видимых; рамка камеры и «сбросить вид» — по union-bbox сборки; размерные линии — по активной.
- **Экспорт**: `mergedExportTris()` — все видимые модели, каждая повёрнута и смещена; один STL с
  несколькими оболочками (слайсеры это штатно едят). В сводке — «Треугольники (экспорт)».
- **Кнопка «Подтвердить»** работает по-прежнему на активной модели; переключение карточки сначала
  коммитит незакреплённые правки (applyPendingNow), чтобы запись сохранила то, что в полях.
- Стабы для Node-тестов расширены (персистентные элементы по id, createElement/appendChild), у
  debounce-теста сохранён алиас fakeStyle.

Проверка: `test_multi_model.js` (33) — снапшот/восстановление параметров-лого-поворота-позиции,
независимость дубля, авто-смещение новой модели, merge экспорта (скрытая модель исключается, поворот и
смещение применяются), union-bbox, удаление активной активирует соседа, последняя модель защищена.
Батарея **457/457 в 17 файлах**. Браузер: сценарий целиком через реальные инпуты/кнопки — добавление,
настройка полого контейнера у модели 2, живое смещение (без пересборки), переключение восстанавливает
поля, merge-STL с корректным числом байт, dup/del, скрытие убирает из сцены и файла; скриншот сборки.

## Этот проход: дебаг-сессия — фаззер нашёл 3 бага, все исправлены

Прогон: батарея (зелёная) + сидированный фаззер (800 случайных комбинаций режимов/фич/логотипов/наклона,
проверка исключений, NaN, водонепроницаемости, знака объёма) + smoke-тест живого приложения в браузере.

**Баг 1 — Т-стык на «прямоугольном» squircle (rim/hollow, squircle≈8 %).** У почти-прямоугольного
суперэллипса три угла угловой ячейки картезианской заливки коллинеарны (лежат на прямом участке кромки
z=hd): одна половинка ячейки — вырожденный треугольник, фильтр его удалял → Т-стык со стенкой (6 открытых
кромок). Фикс: `pushQuadAdaptive` — ячейка делится по той диагонали, где обе половинки невырождены
(max(min area)); применён в плоских крышках, «шапке» купола и гранях cube-sphere.

**Баг 2 — ложные «открытые» кромки на лезвии округления (rounded hollow + сильный наклон).** Две
соседние заплатки скруглённого полого считают одну вершину дуги с разницей ~2e-15; после наклона обе
попадают ровно на границу букета `toFixed(5)` (…0875|) и разъезжаются по разным ключам → manifoldCheck
рапортовал 24 «открытых» кромки на геометрически замкнутом меше (при 7 знаках — 0). Фикс: сварка вершин
в `manifoldCheck` теперь честная допусковая — union-find по пространственному хэшу с радиусом ~полбукета,
не зависит от положения относительно десятичной границы. Семантика (decimals) сохранена.

**Баг 3 — пол 1 мм в клампе логотипа.** При радиусах 11+12 на полом высотой 24 мм плоская полоса стенки —
1 мм, а `clampLogoToFace` принудительно давал логотипу минимум 1 мм (`Math.max(1,…)`) → рельеф вылезал в
цилиндры скругления (4 открытых кромки). Фикс: честный кламп `Math.max(0,…)` — если места нет, логотип
схлопывается в 0 и ничего не рисует (dispFn возвращает 0 — безопасно).

Регрессы закреплены: `test_squircle.js` +4 (боксовые squircle rim/hollow ± squircleV),
`test_fillet_logo_hollow.js` +2 (экстремальные радиусы + микро-лого; наклон без ложных «открытых»).
После фиксов: фаззер **800/800**, батарея **424/424**. Smoke живого приложения: загрузка без ошибок
(кроме заблокированных в песочнице Google Fonts — штатный фолбэк), staged-apply работает (кнопка
подсвечивается, модель не трогается до нажатия, после — пересборка, кнопка гаснет), панель «Меш:
замкнутый ✓», бинарный STL (84+50·n байт, счётчик в шапке) и ASCII STL (solid/facet/endsolid) валидны,
раунд-трип режимов (полый squircle → сплошной) водонепроницаем.

## Этот проход: картезианская «шапка» на полюсе скруглённого дна (логотипы на дне)

Скруглённое дно раньше закрывалось веером у полюса → центральный донный лого дал бы «спицы», поэтому дно
строилось без эмбосса. Теперь веер заменён картезианской «шапкой»: `superellipseDomeFan`→`superellipseDomeCapped`.
- Крутая часть (радиус 1→rCap=0.68) — прежние кольца по высоте.
- Пологий центр (радиус <rCap) — картезианский fill-грид (`g`, отмасштабированный на rCap; высота по профилю
  `yD`). Центр — обычная valence-4 вершина, поэтому донный логотип (`-Y` / `-Y-inner`) embossed вдоль
  численной нормали без «спиц».
- Шов кольцо↔шапка водонепроницаем: граница шапки = `g_boundary*rCap` = переходное кольцо `ring*rCap`
  (общий множитель, точное совпадение), высота обеих = yT.
- Зональное сгущение работает и для шапки (`squircleABYZones` уже сгущает a,b под `-Y`/`-Y-inner`).

Проверка: рег-тесты — донный лого на скруглённом дне водонепроницаем, без веера (valence≤12) и реально
рельефит (треугольников больше, чем без лого); то же для пола полости. Браузер: череп-эмблема на скруглённом
дне контейнера — чисто, без спиц. Батарея **418/418**.

## Этот проход: элипсоидность ко дну контейнера (squircleV скругляет дно)

Раньше в полом/борте squircleV игнорировался (плоское дно). Теперь он скругляет ДНО контейнера как нижнюю
половину супереллипсоида: прямые стенки в верхней половине [0,hh], куполообразное дно в нижней [-hh,0]; 0 —
плоское дно (как было), 100 — глубокая чаша. Верх остаётся открытым с плоским бортом.

- Дно строит `superellipseDomeFan(ring, N, poleY, invEv, sign, nRings)`: концентрические супереллипс-кольца,
  сэмплированные по ВЫСОТЕ y (важно: сэмплинг по y ровно распределяет точки по профилю — и почти
  вертикальный участок у y=0, и пологий у полюса; сэмплинг по горизонтальному радиусу, как в первой попытке,
  недосэмплировал крутую часть и уводил границу с y=0 → шов). Горизонтальный масштаб
  `rS(y)=(1-(y/poleY)^(2/eV))^(eV/2)`. Уровень 0 при y=0 = нижнее кольцо стенки (общие вершины → шов
  закрыт), вниз до полюса с веером.
- Полый (innerDome): полость повторяет купол (внутренний полюс на floorY=-hh+t → толщина стенки t). Борт:
  внешнее дно скругляется, неглубокий карман сверху остаётся плоским (`superellipseFillCap`).
- `buildSquircleHollow` получил параметры `eV` и `innerDome`; при eV≤0 — прежний плоский путь без изменений
  (регрессы целы). Маршруты полого/борта теперь передают eV из squircleV.

Ограничение: логотип на самом скруглённом дне (`-Y` / `-Y-inner`) не проецируется — центральный лого попал
бы на полюсной веер (спицы); дно строится без эмбосса. Боковые лого на стенках работают.

Проверка: `test_squircle.js` +19 — водонепроницаемость по squircleV 20..100 (полый и борт), «скруглённое
дно убирает материал» (объём меньше плоского), «меш меняется», не-куб, полный 100/100, наклон, лого на
стенках. Браузер: контейнер со скруглённым дном (40 % — мягкая фаска снизу, 100 % — глубокая чаша),
водонепроницаемо. Батарея **413/413**.

## Этот проход: зональное сгущение сетки squircle-логотипов (дёшево при 500)

Равномерная сетка при высокой детализации тратит треугольники на всю поверхность, хотя лого занимает
маленький кусок. Теперь сетка ГРУБАЯ везде и МЕЛКАЯ только под лого (та же схема, что у зонального грида
плоского куба).

- `zonedSamples(lo,hi,coarseStep,fineStep,zones)` — грубые сэмплы везде + мелкие в интервалах-зонах (с
  потолком на зону, чтобы огромный футпринт не взорвал число точек).
- `squircleABYZones(p,tInner)` — из футпринта каждого лого считает зоны по осям: крышечный лого (+Y/−Y/пол
  полости) сгущает a и b; боковой — свою ось периметра (a или b, чтобы граничное кольцо было мелким там) и
  высоту y. Нормированные a=x/hw, b=z/hd в [−1,1]; y в мм.
- `superellipseFillGrid` теперь принимает явные массивы `aSamp×bSamp` (не квадрат G×G); `fillGridBoundaryLoop`
  и `superellipseFillCap` работают с прямоугольным гридом. Внешняя и внутренняя оболочки полого делят ОДНИ
  a,b-массивы, поэтому граничные кольца совпадают по числу точек (борт-кольцо сшивается).
- `squircleWall` берёт явный `ySamp` вместо `Mv` (мелко по высоте под боковым лого, грубо у полости).
- Супереллипсоид (cube-sphere): ГЛОБАЛЬНЫЕ по осям cube-координаты (общие для всех 6 граней → каждое общее
  ребро семплится одинаково → водонепроницаемо). Зоны находятся грубым forward-hit-тестом: где
  `applySurfaceLogo` сместил вершину — туда попадает интервал сгущения по двум cube-осям грани.
- Нижний порог мелкого шага `fineW` 0.05→0.1 мм (при 500 это и так тоньше пикселя heightmap-а, так что
  визуально не хуже, а треугольников вдвое меньше).

Замер (браузер, лого на +Y, детализация 500): супереллипсоид 22-мм лого — **319k** тр., ~1.0 с (было ~811k,
2.5 с равномерно); призма 473k; полый 575k. Маленький 12-мм лого — 280k; без лого грубая оболочка — 14k.
Та же чёткость (череп на верхушке — гладкие глазницы, свод, челюсть). Регресс-тест: маленький лого даёт
существенно меньше треугольников, чем полногранный (`small < big*0.7`) — и на крышке, и на боку. Батарея
**396/396**; водонепроницаемость и valence≤12 при высокой детализации сохраняются.

## Этот проход: детализация логотипов до 500

«Увеличь детализацию до 500 во всех случаях». Ограничивали ТРИ вещи, а не только ползунок:
- ползунки «Детализация» (`max=300`) и их `change`-обработчики (`Math.min(300, v)`) → подняты до **500**;
- внутренние потолки сетки squircle-билдеров (cube-sphere `S≤176`, призма `G≤220`, полый `G≤220`) —
  подняты до `S≤260`, призма `G≤260`, полый `G≤200` (полый чуть ниже — две оболочки, тяжелее);
- нижний порог ячейки `spacing` 0.15→0.05 мм, чтобы мелкие объекты тоже дотягивались до высокой детализации.

Замер (браузер, 40-мм объект, лого на +Y): дефолт `res=50` — 120k тр., ~0.33 с; максимум `res=500` —
супереллипсоид ~811k тр. ~2.5 с, призма ~803k ~2.3 с, полый ~981k ~2.5 с. Перегенерация одноразовая (по

проблем. Дефолт не тронут (быстрый). Череп на верхушке супереллипсоида при 500 — гладкие круглые
глазницы и свод, ступенек нет.

Тест-гард на поднятые потолки: сборка при `logoResolution=300` (упирается в `S=260`, ~0.7 млн тр.) —
водонепроницаемо и без веера (valence ≤12). Логотипные тесты гоняются на облегчённом гриде (density-

## Этот проход: плотнее сетка эмбосса логотипов на squircle (глаже края)

Логотип на squircle/супереллипсоиде оставался ступенчатым: эмбосс-грид (`spacing = maxDim/logoRes`, ~50
сэмплов на грань) сильно грубее heightmap-а (256², т.е. ~0.09 мм/пиксель на лого 20 мм). Плюс крышки были
жёстко ограничены потолками (супереллипсоид `S≤96`, призма/полый `G≤120`), так что поднятие ползунка
«Детализация» переставало помогать.

- Базовая плотность ×2: `spacing = max(0.15, maxDim/(logoRes*2))` во всех трёх билдерах (призма,
  cube-sphere супереллипсоид, полый/борт) — вдвое мельче ячейка, вчетверо больше треугольников, края лого
  вдвое глаже.
- Потолки подняты (cube-sphere `S` 96→176, `G` призмы/полого 120→220), поэтому ползунок «Детализация»
  (до 300) снова работает — на максимуме ячейка ~0.2–0.35 мм.
- Свойства меша не изменились (водонепроницаемость, valence, знаки объёма — density-независимы), поэтому
  логотипные тесты гоняются на облегчённом гриде (`logoResolution=24` в `test_squircle.js`), чтобы батарея
  не тормозила; приложение по умолчанию плотнее. Батарея **392/392**. Браузер: тот же череп на верхушке
  супереллипсоида заметно чётче (круглее глазницы, глаже свод/челюсть).

## Этот проход: фикс искажения логотипов на верхушке/дне (полюс/крышка)

Баг (со скриншотом от пользователя): логотип на ВЕРХУ супереллипсоида размазывался в радиальные «спицы».
Причина — топология сетки на полюсах/крышках: и супереллипсоид (lat/long кольца), и плоские крышки призмы
/полого (радиальный диск с веером из центра) СХОДЯТСЯ в одну апексную вершину. Эмбосс детального высотного
поля на веер из центра → лучевые спицы. Стенки (боковой прямоугольный грид) были в порядке — проблема только
на «шапках».

Исправление — убрать полюсную/веерную топологию там, где садится логотип:
- **Супереллипсоид → cube-sphere.** Вместо колец полюс-в-полюс поверхность строится как 6 «раздутых»
  граней куба, каждая — регулярный картезианский патч. Направление грани куба точно проецируется на
  супереллипсоид неявной формулой `F(t·d)=1 ⇒ t=G(d)^(−eV/2)` (`seImplicitSurface`). Верх/низ — обычные
  valence-4 области, без сингулярности. Нормали АНАЛИТИЧЕСКИЕ (`seImplicitNormal`, ∇F) — функция только от
  точки, поэтому вершина на общем ребре двух граней получает одинаковую нормаль (и одинаковый эмбосс) с обеих
  сторон → водонепроницаемо даже когда лого пересекает ребро куба.
- **Плоские крышки (призма верх/низ, полый: внешнее дно + пол полости) → картезианская заливка.**
  `superellipseFillGrid`: единичный квадрат отображается на диск (elliptical grid mapping), затем растягивается
  до границы-супереллипса — центр изотропно-по-осям (лого не искажается), граница — точный супереллипс, а
  ЦЕНТР — обычная valence-4 вершина, без веера. Граница заливки = периметр `fillGridBoundaryLoop`, и его же
  экструдирует стенка, поэтому шов сторона↔крышка закрыт. Радиальный `squircleDiscCap` и uniform-θ
  `squircleRingAndNormals` удалены.

Регресс-тест прямо на баг: `maxVertexValence(tris)` — у веерной вершины валентность ≈ N (число сегментов), у
картезианского/cube-sphere грида ~6. Проверки требуют ≤12 для логотипов на +Y/−Y призмы, полюсах
супереллипсоида, дне и полу полости. `test_squircle.js` 77→86, вся батарея **392/392**. Браузер: череп на
верхушке супереллипсоида и на плоской крышке призмы — читается чисто (свод, глазницы, нос, челюсть), без спиц,
`manifoldCheck` watertight.

## Этот проход: squircle как ПОЛЫЙ контейнер и БОРТ/лоток

Задача: «вынеси [squircle] на полый/борт». Раньше ветка `p.squircle>0` стояла ПОСЛЕ веток `p.hollow`/
`p.rim` в `buildTrisForShape`, поэтому срабатывала только на сплошном кубе. Теперь squircle проверяется
в САМОМ НАЧАЛЕ обеих веток и перекрывает филлет/сетчатое дно (squircle — это и есть обработка угла).

Общий билдер `buildSquircleHollow(w,h,d,p,t,floorY,...)` — оболочка из супереллипса:
- внешняя стенка (супереллипс `hw,hd`, y −hh..hh) — `squircleWall(..., sign=+1)`, нормали наружу;
- полость (супереллипс `hw−t,hd−t`, y floorY..hh) — `squircleWall(..., sign=-1)`, нормали в пустоту;
- внешнее дно (диск `hw,hd` при −hh, нормаль −Y) и дно полости (диск `hw−t,hd−t` при floorY, +Y) —
  `squircleDiscCap` (радиальная сетка концентрических супереллипсов);
- верхний борт — плоское кольцо-аннулюс между внешним и внутренним верхними кольцами.
Один и тот же билдер обслуживает оба режима, отличается только `floorY`: полый = −hh+t (тонкое дно у
низа), борт = hh−rimHeight (неглубокий карман сверху, толстая база снизу).

Толщина стенки: внутренний супереллипс = внешний, ужатый по габаритам на `t` (тот же показатель) —
толщина ≈ t на «плоских» участках и слегка гуляет на углах (та же оговорка, что у бокс-полого). Верх/низ
рёбра НЕ скругляются (squircleV игнорируется) — открытый контейнер стоит на плоском дне и имеет плоский
борт. Водонепроницаемо: каждая поверхность — замкнутый грид, а швы (стенка↔крышка, стенка↔борт) лежат на
несмещённых крайних кольцах, поэтому эмбосс логотипов их не трогает.

Логотипы: внешние стенки embossed через `applySurfaceLogo` (грань по доминантной оси нормали, как у
сплошного squircle); внешнее дно — по `dispFns['-Y']` вдоль −Y; дно полости/кармана — по
`dispFns['-Y-inner']` вдоль +Y (обе крышки — радиальные диски, эмбосс = 0 на границе через кламп). Стенки
полости и борт-кольцо — без рельефа. Наклон применяется после (`taperBoxTris`), фреймы в плоском
пространстве — та же конвенция.

Проверка: `test_squircle.js` расширен 50→77 (полый: роундность, толщина стенки, не-куб, наклон, «полость
< сплошного», «squircleV игнорируется»; борт: роундность, rimHeight, наклон, «борт > глубокого полого»;
логотипы на стенках/дне/полу полости, гравировка, два лого, лого+наклон). Вся батарея **383/383**. Браузер:
полый squircle-контейнер (открытый верх, тонкие стенки, плоский борт), борт с неглубоким карманом, лого
оборачивается по изогнутой стенке — `manifoldCheck` в живом приложении watertight.

**На будущее:** сетчатого дна на squircle-полом нет (там другой билдер дна). squircleV на контейнере не
поддержан (открытый верх + скруглённый = странно).

## Этот проход: логотипы на squircle / супереллипсоиде (emboss на изогнутую стенку)

Задача: «сделай совместимость логотипов с последними нововведениями» — т.е. с squircle-призмой и
супереллипсоидом. Раньше маршрут `p.squircle>0` строил гладкую оболочку и логотипы игнорировал (плоская
грань, на которую садится рельеф, там отсутствует).

Механика: логотипный `dispFn` (`makeLogoDispFn`) проецирует 3D-точку на касательный фрейм лого и
возвращает глубину — это ровно то, что нужно кривой стенке, он просто «оборачивает» рельеф по изгибу.
Поэтому вместо плоской грани оболочка строится МЕЛКОЙ (u,v)-сеткой, и в каждой вершине:
- грань выбирается cube-map-стилем по доминантной оси внешней нормали (`applySurfaceLogo`), так что
  вершина берёт рельеф не более чем ОДНОЙ грани;
- вершина смещается вдоль ЛОКАЛЬНОЙ внешней нормали на возвращённую глубину (рельеф ⟂ изогнутой стенке).

Каждая вершина имеет одну позицию, общую для всех своих треугольников → меш остаётся водонепроницаемым
(тот же аргумент «высотное поле над сеткой», что у логотипов на плоском кубе). Лого клампится от рёбер
(`clampLogoToFace`), поэтому рельеф = 0 вдоль всех швов (сторона↔крышка, полюс) — швы закрыты.

- `buildSquirclePrismLogos(w,h,d,p,seg,dispFns,logoRes)`: мелкая боковая сетка (периметр × высота) +
  крышки радиальной сеткой (концентрические супереллипсы), так что лого на +Y/−Y тоже садится на плоскую
  крышку. Верх/низ боковых колец держатся на базовом кольце (рельеф там 0), поэтому точно совпадают с
  граничным кольцом каждой крышки → шов закрыт.
- `buildSuperellipsoidLogos(...)`: одна сетка колец полюс-в-полюс; нормали — центральной разностью
  (полюса как соседи по φ); каждая вершина embossed и смещена вдоль нормали. Полюс — одна вершина.
- Плотность сетки берётся от `logoResolution` (шаг ≈ maxDim/logoRes), с потолками N/M. Путь БЕЗ логотипов
  не тронут (быстрый веерный билдер) — старые проверки объёма/числа треугольников без изменений.
- Маршрут в `buildTrisForShape` (ветка `p.squircle>0`): фреймы строятся в плоском (identity-taper)
  пространстве, наклон применяется после через `taperBoxTris` — та же конвенция, что у fillet+logo.

Проверка: `test_squircle.js` расширен 25→50 (лого на всех гранях призмы + плоские крышки; лого на боках
и полюсе супереллипсоида; гравировка, поворот, наклон, два лого на разных/одной грани, не-куб; регресс
пути без логотипов). Вся батарея **356/356**. Браузер (headless Chromium + Playwright): рельеф-сердечко
embossed на изогнутый бок squircle-призмы, супереллипсоида и два лого на двух боках — визуально
оборачивается по кривизне, `manifoldCheck` в живом приложении = watertight.

**На будущее:** очень широкое лого на боку клампится плоскостью грани, но cube-map-гейт обрезает его на
диагонали 45° (лого не заворачивается за угол) — держать лого по центру грани. Squircle всё ещё только на
СПЛОШНОМ кубе (полый/борт — потом).

## Этот проход: Squircle / супереллипс (то, что имелось в виду под «скруглением стенок»)

Пользователь прислал картинку squircle (иконка iOS) — оказалось, под «скруглением стенок» он имел в
виду СУПЕРЭЛЛИПС, а не подушку-на-грани. Проверка в браузере (headless Chromium + Playwright, Three.js
r128 из npm, локально) подтвердила: выгиб+филлет давали «подушку с горбом на грани» — не то. Добавлен
отдельный билдер.

- `buildSquirclePrism(w,h,d,p,resolution)`: горизонтальное сечение — супереллипс
  `x=a·sign(cosθ)·|cosθ|^p, z=c·sign(sinθ)·|sinθ|^p`, экструзия по высоте, плоские верх/низ (веерные
  крышки). `p∈(0,1]`: 1 — эллипс, ~0.5 — squircle, →0 — прямоугольник. Обмотка через `orient` по
  радиальной нормали; вырожденные треугольники отбрасываются.
- UI-параметр `squircle` 0..100 % (0=off/прямоугольник, 100=эллипс), `p=squircle/100`. Маршрут
  `buildTrisForShape` (сплошной): `p.squircle>0` → `buildSquirclePrism`, наклон поверх
  (`taperBoxTris`). Перекрывает скругление углов и выгиб стенок (это другая трактовка «скругления»).
- Браузерная проверка: `verify.html` (копия с локальным Three.js), Playwright-драйвер с SwiftShader
  (`--use-angle=swiftshader`), скриншоты сверху (сечение = супереллипс, 1-в-1 картинка) и 3D.

Проверка: `test_squircle.js` 16/16 через реальный `buildTrisForShape` — водонепроницаемость на
роундности 8..100 %, не-куб, наклон, знак объёма>0, монотонность (эллипс<квадрат по объёму), squircle
перекрывает филлет, регресс squircle=0 → обычный бокс (12 треугольников). Вся батарея 322/322.

**На будущее:** squircle только на СПЛОШНОМ кубе, верх/низ плоские (можно добавить вертикальное
скругление рёбер → супереллипсоид); логотипы на кривом боку пока не проецируются; полый/борт — потом.

## Этот проход: выгиб стенок повторяется полостью (толщина стенки сохраняется)

Раньше выгиб ехал только по ВНЕШНЕЙ оболочке полого — полость оставалась прямой, стенка утолщалась в
месте выгиба (и вогнутое ограничивалось толщиной стенки). Теперь **внутренняя полость повторяет выгиб**,
двигаясь в ту же мировую сторону, что и внешняя стенка → толщина сохраняется.

- Общий примитив `makeRectDome(amp,uLo,uHi,vLo,vHi)` (купол над произвольным прямоугольником, 0 на
  границе); `makeInsetBulgeDisp` теперь его частный случай.
- **Скруглённый полый/борт:** `buildAsymBoxSurfaces` применяет инсет-купол к ОБЕИМ оболочкам с ОДНИМ
  знаком (не инвертируя, в отличие от логотипов) — внутренняя грань двигается в ту же сторону, что и
  внешняя. Проброшено через `buildRoundedHollow`/`buildSharpRimHollow` (и на внешний, и на внутренний
  вызов). Зоны выгиба теперь ВЛИВАЮТСЯ в `zonesByAxis` (`mergeZones`), иначе середина грани не
  подразбивалась и купол выходил гранёным.
- **Плоский полый:** `buildHollowBox` получил `bulgeAmpsByFace`; 4 стенки полости и дно получают
  `cav()`-купол (знак инвертирован, т.к. нормаль полости смотрит внутрь). Y полости подразбивается
  (`bulgeSubdivRange`), иначе 2-строчная полоса не показала бы купол (он 0 на концах).
- **Борт/лоток:** карман мелкий у самого верха, где купол внешней стенки ≈0, поэтому карман НЕ выгибаем
  (независимый купол там был бы неверным); плоский лоток — только внешние стенки.
- Вогнутый клампится по размеру ПОЛОСТИ (`0.8·(min(hw,hh,hd)−t)`), а не толщине стенки — полость теперь
  тоже двигается, поэтому предел мягче; ограничение — чтобы стенки полости не пересекли центр.

Проверка: `test_wall_bulge.js` 53/53 — добавлено «полость повторяет выгиб» (замер: внутренняя стенка
уехала с 16 на ~22 при внешней на ~26 → зазор = толщине; для плоского/скруглённого/острого полого),
знак объёма > 0, вогнутый под клампом. Регресс — вся батарея `./run-all.sh` 306/306.

## Этот проход: скругление стенок — фаза 2 (со скруглением углов + на полом/борте)

Выгиб граней доведён до полной совместимости. Две части:

1. **Выгиб + скругление углов (сплошной куб).** Купол теперь строится в ИНСЕТ-версии
   (`makeInsetBulgeDisp`): тот же приподнятый косинус, но центрированный и масштабированный на
   плоскую ИНСЕТ-зону грани (между скруглёнными рёбрами). Т.к. `wallBulgeProfile(|x|≥1)=0`, купол
   ноль и на границе плоской зоны, и в угловых «расширениях» → шов с цилиндрами рёбер цел.
   `buildAsymRoundedBox` получил параметр `bulgeAmpsByFace`: в цикле граней инсет-купол СКЛАДЫВАЕТСЯ
   с логотип-dispFn. Маршрут: при `hasWallBulge && hasFillet` → `buildAsymRoundedBox` с выгибом +
   лого(flat) + объединёнными зонами, наклон поверх (`taperBoxTris`).
2. **Выгиб на полом/борте (внешние стенки + дно).** `buildAsymBoxSurfaces` получил `bulgeAmpsByFace`
   (инсет-купол, ТОЛЬКО внешняя оболочка `!flip`); проброшен через `buildRoundedHollow`/
   `buildSharpRimHollow`. Для ПЛОСКОГО полого/борта выгиб просто вливается в per-face dispFns
   (`buildWallBulgeDispFns`), т.к. их внешние грани уже полностью сетчатые. Полость остаётся ПРЯМОЙ →
   вогнутый выгиб клампится по толщине стенки (`wallBulgeAmounts(p, 0.8·t)`), чтобы не прорезать полость.
   Детализация: у полого/борта — `logoResolutionHollow`.

Хелперы обобщены: `wallBulgeAmounts(p, concaveCap)`, `buildWallBulgeDispFns(p, amps)`,
`computeWallBulgeZones(p, amps)`. UI-подпись обновлена (работает во всех режимах, сочетается со
скруглением углов). Убран приоритет-над-филлетом из фазы 1.

Проверка: `test_wall_bulge.js` 41/41 — добавлены выгиб+филлет (в т.ч. асим-радиусы, +логотип на той же
грани, +наклон; комбо двигает bbox наружу, рёбра остаются скруглёнными) и все ЧЕТЫРЕ полых/бортовых
билдера (плоский полый, скруглённый полый, острый-борт, плоский борт, скруглённый борт) —
водонепроницаемо. Прото инсет-купола на реальном `buildAsymRoundedBox` — 5/5. Регресс — вся батарея
`./run-all.sh` 294/294.

## Этот проход: скругление стенок (per-face bulge) + совместимость с логотипами

Добавлен ВЫГИБ граней сплошного куба — отдельно от скругления углов (рёбер). Каждая из 6 граней
получает знаковый купол: `+` наружу («бочка»), `−` внутрь («подушка»), 0 — плоско. Ключевое
наблюдение: выгиб грани = ровно тот же per-face displacement, на котором едут логотипы, только на
всю грань, поэтому весь механизм переиспользован и **совместимость с логотипами получается сложением
смещений** (не «или-или»).

- Профиль: сепарабельный купол `amp·B(u/uH)·B(v/vH)`, `B` — приподнятый косинус (`(1+cos πx)/2`),
  1 в центре, **0 на всех четырёх рёбрах грани**. Именно 0-на-рёбрах гарантирует, что швы с соседними
  гранями и с существующим скруглением углов не рвутся — поэтому выбран купол (а не «ствол»/цилиндр
  по одной оси, который сдвигал бы общие рёбра и потребовал бы переделки соседей).
- Новые функции: `WALL_BULGE_FACES`/`WALL_BULGE_KEYS`, `wallBulgeProfile`, `wallBulgeAmounts`
  (клампы: вогнутое `≥ −0.45·размер_по_нормали`, чтобы противоположные стенки не пересеклись),
  `buildWallBulgeDispFns`, `computeWallBulgeZones` (полноосевые зоны → мелкая сетка на всю грань),
  `mergeDispFnsAdd`/`mergeZones`. Маршрут `buildTrisForShape` (сплошной): при `hasWallBulge` →
  `buildBoxWithLogos` с суммой (лого+выгиб) dispFns и объединёнными зонами; наклон применяется внутри.
- UI: 6 знаковых полей «Скругление стенки ±X/±Z», «верха +Y», «дна −Y» (−150..150 мм). Детализация
  купола берётся из «Детализация (сплошной куб)» (`logoResolution`).
- **Границы (на будущее):** только сплошной куб; в приоритете над скруглением углов (одновременно не
  сочетаются — нужен купол с 0 на инсет-границе филлета); полый/борт — следующая фаза (как было с
  логотипами и филлетом).

Проверка: `test_wall_bulge.js` 24/24 через реальный `buildTrisForShape` — водонепроницаемость
(выпукло/вогнуто/смешанные знаки/не-куб/экстрим-кламп), направление знака (выпуклое двигает bbox
наружу, вогнутое оставляет рёбра на месте), выгиб+наклон, выгиб+логотип на одной грани (пик =
hw+выгиб+глубина лого — рельеф действительно складывается), приоритет над филлетом, регресс «без
выгиба = 12 треугольников». Прото на реальном `buildBoxWithLogos` — 10/10. Регресс — вся батарея
`./run-all.sh` 277/277.

## Этот проход: сетчатое диагональное дно полого контейнера (садовая решётка)

Дно ПЛОСКОГО полого контейнера превращается в **диагональную решётку** (садовая решётка): чистые
рёбра ±45°, ромбовидные сквозные отверстия, плоский профиль. Флажок `latticeFloor` + `latticeCell`
(шаг ромба), `latticeRib` (толщина ребра), `latticeBorder` (сплошной кант в ячейках).

Итерация: первая версия была «ромб, вписанный в квадратную ячейку» — но это читалось как перфорированная
плита (есть и ортогональные рёбра, плоские узлы). Пользователь показал фото садовой решётки → нужен
ЧИСТО диагональный узор. Переделал билдер (выбор: плоский профиль + чистая диагональ).

Строится на **той же мелкой сетке `xMid×zMid`**, что уже использует дно полости и её стенки
(`midSlice()` включает концы → `xMid = xPositions ∩ [-(hw-t),hw-t]`), поэтому все швы совпадают со
стенками/кольцом без конфликта разбиения. Воксельный подход:
- Каждая мелкая ячейка = СПЛОШНАЯ (часть ребра) или ДЫРА. Принадлежность ребру — в повёрнутых координатах
  `uu=(X+Z)/pitch`, `vv=(X−Z)/pitch`: ячейка сплошная, если в пределах `rib/2` (world) от целой линии `uu`
  или `vv` — т.е. лежит на одном из двух семейств ±45°. Рёбра идут по 45°, сетка осевая → границы падают
  на рёбра ячеек; выше разрешение сборки — глаже диагональ.
- **Сплошная ячейка**: верхний квад (дно полости, +Y) + нижний квад (внешнее дно, −Y) + вертикальные
  стенки-рёбра там, где сосед — дыра (нормаль внутрь дыры). Кант (`border`, минимум 1) — сплошной, дыры
  не доходят до стенок; «снаружи региона» = сплошное, поэтому границу закрывает сам контейнер (без стенки
  по периметру решётки).
- **Устранение защемлений (pinch)**: тонкое ребро = ячейки, касающиеся только углом (2×2 шахматка) —
  негерметично (вертикальное ребро используется 4 раза). Проход заливает одну дыру каждой шахматки —
  мостит их (и скругляет узлы пересечения). Монотонно (только добавляет сплошное) → сходится.
- **Внешнее дно** эмитится как **КОЛЬЦО** (ячейки `xP×zP` вне внутреннего прямоугольника); плоская
  полнофутпринтная грань `−Y` больше не строится.

Обмотка берётся на ПЛОСКИХ координатах (`orient` до `taperFn`), затем наклон — швы совпадают с
наклонёнными стенками/кольцом. **`pitch`/`rib` заданы в мм и НЕ зависят от разрешения** — размер
отверстия расцеплён с детализацией стенок/логотипов (стенки на постоянном `logoResolutionHollow`).
Решётка не несёт рельеф — логотипы на `−Y`/`−Y-изнутри` не рисуются, на стенках работают как обычно.

Проверка (Node, реальный код через `buildTrisForShape`): `test_lattice_floor.js` 26/26 —
водонепроницаемо на размерах/шаге/ребре/канте, **тонкое ребро (путь pinch-removal)**, наклон стенок,
логотипы на стенках, регресс «полый без решётки», расцепление (число треугольников стенок побайтно
одинаково при разном `cell`). Прототипы: `hollow-diagnet.js` 11/11 (диагональ+pinch+наклон),
`diag-net-v3.js` 8/8 (автономная плита), ранние `hollow-lattice-v2.js` 17/17 / `v1` 18/18 (ромб-в-квадрате,
заменён). Браузер: полый 50×36×50, шаг 9, ребро 1.6 — watertight (0 открытых/битых), маска дна = чистая
диагональная решётка (ромбы, узлы-пересечения, сплошной кант).

**Ограничения (на будущее):** решётка пока только на ПЛОСКОМ полом (не на скруглённом — там дно строит
другой билдер) и не на борте/лотке. Плоский профиль ребра (не круглый). Диагональ имеет мелкую «лесенку»
по краям (шаг = мелкая ячейка сетки; чем выше `Детализация полый контейнер`, тем глаже).

## Этот проход: логотипы на скруглённом ПОЛОМ/БОРТЕ (Фаза 2)

Логотипы поехали на скруглённый полый контейнер и борт: **рельеф на внешних стенках + дно полости/
кармана**. Переиспользована машинерия Фазы 1:

- `buildAsymBoxSurfaces` получил `dispFnsByFace`: раздаёт логотип-dispFn по граням оболочки. Для
  ВНУТРЕННЕЙ оболочки (flip=true) смещение инвертируется — +глубина стоит ВЫПУКЛО в полость (как у
  внешних стенок). Грани строятся в ЛОКАЛЬНЫХ Y-координатах (yOffset применяется после), а фрейм
  логотипа — в world, поэтому перед проекцией к вершине добавляется yOffset.
- `buildRoundedHollow` / `buildSharpRimHollow` приняли `dispFnsOuter`/`dispFnsInner` + зоны + logoRes.
  Зоны загущают общие `xP/zP` (world) и `yPo` (для sharp — со сдвигом на −ciyO, т.к. оболочка поднята).
  Общий хелпер `addLogoZonesToAxis` (вынесен из `buildAsymRoundedBox`).
- Маршрут `buildTrisForShape` (hollow/rim): условие скругления больше не требует «нет логотипов»;
  `splitHollowLogoDispFns()` делит логотипы на внешние грани (→ outer) и дно полости `-Y-inner`
  (→ inner как грань `-Y`), строит плоские зоны. Плоский полый (без скругления) с логотипами
  по-прежнему идёт в `buildHollowBox`/`buildRimBox`.
- **Кламп** (`faceFilletInsets` расширен на hollow/rim): внешние стенки клампятся по внешним радиусам
  (+ верхний инсет: rolled lip `min(cLip,0.9t)` или у sharp — до `y=hh`), дно полости — по внутренним
  (`max(iFloor, iVert)`). `LOGO_CLAMP_KEYS` дополнен внутренними радиусами, толщиной стенки, высотой борта.

Проверка: `test_fillet_logo_hollow.js` 19/19 (стенки/дно, полый+борт, гравировка, поворот, rolled/sharp,
стена+дно, кламп oversize на всех гранях). Регрессы: fillet-logo(solid) 16/16, rounded-fillet 14/14,
rim 22/22, hollow-taper 19/19, hollow-inner-logo 12/12, resolution 9/9, e2e 21/21, logo-zone 43/43.
Браузер: скруглённый полый куб 60 + кольцо-логотип на внешней стенке — watertight, рельеф в плоской
зоне между скруглёнными рёбрами.

**Итог по совместимости скругления и логотипов:** сплошной куб, полый контейнер, борт/лоток — во всех
логотипы совместимы со скруглением. Наклон+лого+скругление: рельеф слегка тянется наклоном (принято).

## Этот проход: совместимость скругления и логотипов (Фаза 1 — сплошной куб)

Раньше скругление и логотипы были взаимоисключающими. Теперь **на сплошном скруглённом кубе логотип
рельефом ложится на плоские грани**. Как:

- `buildFaceAsymFromAxis` получил параметр `dispFn`: вершины плоской грани смещаются вдоль (постоянной)
  нормали грани на значение хитмапа. Обмотка берётся от ПЛОСКОГО треугольника (orient до смещения), потом
  смещение — стенки рельефа наследуют согласованную обмотку. Общая вершина сетки получает одинаковое
  смещение в каждой ячейке (dispFn — чистая функция u,v) → рельеф герметичен.
- `buildAsymRoundedBox` принимает `dispFnsByFace` + `zonesByAxis` + `logoRes`: в зоне логотипа общие
  массивы позиций дозагущаются (`logoRes` подразбиений на зону, как sharp-box `buildLogoAwareAxisPositions`),
  каждой грани раздаётся её dispFn.
- **Кламп** (`faceFilletInsets`): плоская зона грани сжата на радиус смежного скругления; логотип
  клампится внутрь (у краёв dispFn=0 → шов с цилиндрами рёбер цел). Перекламп всех логотипов при
  изменении размеров/радиусов/режима (`LOGO_CLAMP_KEYS` + слушатели инпутов).
- Логотип-фрейм в fillet-пути строится во ФЛЭТ-пространстве (identity taper), т.к. наклон применяется
  ПОСЛЕ (`taperBoxTris`) ко всему мешу. При наклон+лого+скругление рельеф слегка растягивается наклоном
  (принято); без наклона — точно.

Маршрут `buildTrisForShape` (сплошной куб): логотипы без скругления → старый `buildBoxWithLogos`;
логотипы + скругление → `buildAsymRoundedBox` с dispFns (uniform радиусы тоже идут сюда, т.к.
`genRoundedBox` рельеф не несёт). Проверка: `test_fillet_logo.js` 16/16 (гравировка, асим/большой/uniform
fillet, все грани, поворот, наклон, 2 лого, кламп размера/позиции, перекламп после увеличения радиуса);
регрессы `test_logo_zone` 43/43, `e2e` 21/21, `rounded_fillet` 14/14, `rim` 22/22. Браузер: скруглённый
куб r8 + кольцо-логотип на +Z — watertight, рельеф на плоской грани, рёбра скруглены.

**Не сделано (Фаза 2):** логотипы на скруглённом ПОЛОМ/БОРТЕ (внешние стенки + дно полости) —
там логотип пока по-прежнему сбрасывает скругление.

## Этот проход: ОСТРЫЙ плоский внутренний кант (без шпилей и без закругления)

Предыдущий проход убирал шпили, ЗАКРУГЛЯЯ нулевой кант (0 → радиус вертикали). Пользователь хотел
другое: **острый плоский кант** — проём со скруглёнными вертикальными углами, но ОСТРЫМ верхним ребром,
без конусов. Chamfer-машинерия так не умеет (она привязывает форму проёма к филлету), поэтому добавлен
отдельный билдер `buildSharpRimHollow` (порт из `scratchpad/asym-sharp-rim-v8.js`):

- **Внутренняя полость**: полностью скруглённый бокс, чей ВЕРХ ПОДНЯТ на `iv` (радиус вертикали) — так
  верхний филлет оказывается ВЫШЕ плоскости проёма `y=hh`; затем все треугольники выше `hh` срезаются.
  Срез оставляет вертикальные углы постоянного радиуса (дуги) и прямые верхние рёбра стенок →
  проём = скруглённо-прямоугольный с ОСТРЫМ ребром.
- **Борт**: явное плоское кольцо между граничной петлёй внешней оболочки и петлёй проёма полости. Обе
  петли — реальные извлечённые рёбра (`sharpTopBoundaryLoops`), их вершины общие → швы герметичны.
  Триангуляция кольца: угловая развёртка (`sharpAnnulusSweep`), при самопересечении — фолбэк на
  лофт «короткой диагонали» с перебором старта (`sharpAnnulusLoft`). Развёртка и лофт всегда
  герметичны (переиспользуют вершины петель без «моста»), различаются лишь самопересечением плоского
  кольца — их проблемные случаи разные, объединение закрывает практически всё.

Маршрут в `buildTrisForShape` (полый и борт): `iLip>0` → `buildRoundedHollow` (закатанный кант, как
раньше); `iLip==0` → `buildSharpRimHollow` (острый). UI-подпись: «(0=острый плоский)».

**Доработка (убраны «рожки» на верхних углах):** в остром режиме ВНЕШНЯЯ оболочка тоже получила
острый-плоский верх (тот же raise-truncate). Раньше внешний закатанный кант зажимался до `0.9·t`
(на тонкой стенке — крошечный), а вертикали 10мм на верхнем углу сужались к нему → выступающие
«рожки». Теперь внешние вертикали держат полный радиус до обода (без сужения), верх — острое ребро,
углы гладкие. Два клампа для валидности: (1) `iv ≤ глубина_полости·0.95` — иначе поднятый бокс
полувысоты `(iv+глубина)/2` не вмещает вертикальный филлет (мелкий трей вырождался); (2) `ov ≤ iv+t`
— иначе большой внешний угловой радиус прорезает за стенку («съедает» её на углу). Для совпадающих
радиусов (обычный случай) оба клампа — no-op. Стресс поднялся до **2000/2000** (watertight И
ориентация). Итог: острый режим = весь верх острый (оба ребра); скруглённый (`iLip>0`) = оба канта
закатаны (внешний `filletTop`).

Стресс `asym-sharp-rim-v8.js` (продукт-клампы): **2000/2000 водонепроницаемы**, 1998/2000 ещё и
ориентационно-согласованы (2 — патология «тонкая стенка + огромные радиусы», герметичны, лишь пара
фасеток борта). Регресс на извлечённом из HTML коде: `test_rounded_fillet.js` 14/14 (кант=0 идёт
острым путём), rim 22/22, hollow-taper 19/19, resolution 9/9, inner-logo 12/12. Браузер: точный кейс
юзера (40³ борт, дно=вертик=10, кант=0) — острый плоский проём, шпилей нет, watertight; сплошной/
полый/борт/тапер — все watertight.

**Замечание:** при рассинхроне первый мой рендер показывал устаревший кадр (`regenerate()` меняет
геометрию синхронно, но canvas перерисовывается на следующем кадре) — не баг билдера.

## Этот проход: убраны «шпили» в кармане + защита от хрупких суб-филлетов

Внутренний кант борта/полости при значении 0 давал конусы-«шпили»: если внутренняя вертикаль
скруглена (>0), а кант = 0, то в верхнем углу `cornerMin = min(vert, 0, 0) = 0`, и вертикальный
цилиндр полости сходится в точку на плоскости борта — тем выше, чем мельче карман. Правка в
`buildRoundedHollow`: **нулевой кант откатывается к радиусу внутренней вертикали** (закатанный
внутренний кант, зеркально закатанному ВНЕШНЕМУ), с клампом к полуглубине кармана `ih/2·0.98`,
чтобы кант не уходил глубже полости. Порог `LIP_EPS=0.1`: если откат вышел бы суб-0.1 (исчезающий),
кант остаётся острым. UI-подпись поля: «(0=как вертикаль)».

Отдельно всплыл ПРЕ-существующий баг (не от этой правки, воспроизвёлся и на старом билдере):
суб-микронная внутренняя вертикаль (напр. 0.0001 мм после клампа) строит вырожденный филлет-осколок,
который `manifoldCheck` при dec5 честно ловит как non-manifold (open+bad). Пользователь такое не
задаёт (шаг UI 0.5), но клампы теоретически могут дожать до этого. Правка в `asymRadiiFromGroups`:
любой радиус группы `< 0.1` после клампа **снапится в 0** (острое ребро вместо хрупкого осколка).
Покрывает сплошной куб, полый и борт, внешнюю и внутреннюю оболочки.

Прототип `scratchpad/asym-hollow-v7.js`: 17 directed + рандом-стресс **2000/2000** водонепроницаемы
(с продукт-клампом, кант принудительно 0 в половине случаев). Регресс на извлечённом из HTML коде:
rim 22/22, hollow-taper 19/19, hollow-resolution 9/9, hollow-inner-logo 12/12, e2e 21/21,
logo-zone 43/43, **новый `test_rounded_fillet.js` 14/14** (ранее путь скругления Node-тестами не
покрывался). В браузере: точный кейс юзера (40³ борт, дно=вертик=10, кант=0) — watertight, карман
чисто скруглён, шпилей нет; сплошной/полый/борт — все watertight.

**Не сделано (на будущее):** острый плоский кант БЕЗ шпилей (постоянный радиус вертикали + плоский
клип + скруглённо-прямоугольный борт) — это крупная переделка ядра asym-билдера; текущее решение —
закатывать кант (как и внешний). Следующий запрошенный шаг: совместимость скругления с логотипами.

## Этот проход: скругление режима «борт»/лоток (rounded tray)

Режим «борт» получил полное скругление. Ключевое наблюдение: **трей топологически = полый
корпус**, только внутренняя полость МЕЛКАЯ (дно кармана на `hh−rimHeight`, а не `−hh+t`;
материал под карманом сплошной). Меш идентичен по структуре (внешняя оболочка + полость-карман +
борт), поэтому `buildRoundedHollow` параметризован глубиной кармана `pocketDepth`: задан → трей
(`ih=pocketDepth`, дно на `hh−pocketDepth`), не задан → полый (`ih=h−t`, дно на `−hh+t`). Ветка
`rim` в `buildTrisForShape` при заданном скруглении и без логотипов зовёт `buildRoundedHollow`
с `pocketDepth=rimHeight`; внутренние радиусы клампятся относительно `rh/2`. Прототип
`scratchpad/asym-tray-v6.js`: стресс 2000/2000 герметичны. Браузер: sharp/outer/full-both-lips/
deep-pocket/taper — все watertight, рендер верный (мелкий скруглённый карман в сплошном боксе).

**Итог по асимметричному скруглению:** сплошной куб (3 группы), полый (внешн.+внутр.+оба канта)
и борт/лоток — все скруглены и водонепроницаемы. Взаимоисключение с логотипами сохраняется.

## Этот проход: скруглённый ВНУТРЕННИЙ кант борта

Полость получила скруглённый верхний кант — исчезли острые «шпили» на внутренних верхних
углах. Ключевой ход: внутренняя полость = вывернутый ВЫПУКЛЫЙ бокс-«пробка», поэтому достаточно
дать внутреннему боксу `rTopIn>0` (внутренний бокс перестал занулять верх) — новой вогнутой
машинерии не нужно. Борт стал КОЛЬЦОМ: `buildAsymRimAnnulus` = внешняя +Y грань минус
треугольники, чей центроид внутри внутреннего проёма (`innerMin` из `asymTopMargins`). Т.к.
общие массивы синхронизированы для обеих оболочек, граница внутреннего проёма (прямые + угловые
шевроны) ложится на рёбра этой грани → центроид-тест режет чисто, внутренний край борта точно
совпадает со скруглённым верхним кантом внутренних стенок. Кламп: внутренний кант `≤` внутр.
вертикали (иначе угол проёма управляется меньшей вертикалью, а борт читает больший инсет →
рассинхрон). UI: поле «скругление внутр. канта» (`filletInnerLip`). Прототип
`scratchpad/asym-hollow-v5.js`: стресс 2000/2000 герметичны. Браузер: inner lip / full both lips
/ innerlip>vert-кламп / taper — все watertight, внутренних «шпилей» нет.

**Не сделано:** режим «борт»/лоток (`buildRimBox`) со скруглением.

## Этот проход: скруглённый ВНЕШНИЙ кант борта (rolled outer lip)

Полый режим получил скруглённый верхний кант снаружи — исчезли острые «шпили» на верхних
внешних углах (раньше вертикальная фаска сходила в острый верх при `rTop=0`). Теперь внешний
бокс сохраняет верхние радиусы (`filletTop` → радиус канта), строит верхние цилиндры/сферы, а
борт-рамка = верхняя грань внешнего бокса (`buildAsymRimFaceWithHole`) С ВЫРЕЗОМ под полость
(треугольники с центроидом внутри `[±ihw,±ihd]` отбрасываются — вырез чистый, т.к. границы
на линиях сетки). Геометрическое условие: кант `< t` (иначе фаска нависает над полостью,
`hw−rTopOut < ihw`, борт исчезает) → кламп `rTopOut ≤ 0.9·t`. Прототип `scratchpad/asym-hollow-v4.js`:
стресс 2000/2000 герметичны. В браузере: lip / lip+внешн+внутр / lip>t-кламп / taper — все watertight.

**Не сделано:** внутренний вогнутый кант (внутренние верхние углы всё ещё дают «шпили» —
полость острая сверху); режим «борт»/лоток (`buildRimBox`) со скруглением.

## Этот проход: скругление полого контейнера (внешние + внутренние рёбра)

Полый режим получил скругление, интегрировано в HTML. Композиция переиспользует asym-примитивы:
внешняя оболочка (скруглённые низ+вертикали, открытый острый верх) + внутренняя полость
(скруглённые пол+вертикали, вывернутые нормали, дно на `−hh+t`) делят ОДИН общий массив X/Z
(+ по-оболочечные Y), плоская борт-рамка при `y=hh` их соединяет. Ключевое: diagonal-sync
обеих радиусных схем гоняется до общей неподвижной точки (внутренняя добавляет точки в общий
X/Z → внешнюю надо пересинхронить, и наоборот).

Новые функции в `<script>`: `buildRoundedHollow`, `buildAsymBoxSurfaces`, `buildAsymFlatRim`,
`asymSyncDiagonalPair`/`asymEdgeSpecs`/`asymAllRadii`. UI: 2 поля «скругление дна/вертик.
полости». Ветка `buildTrisForShape` для `hollow`: если задано скругление и нет логотипов →
`buildRoundedHollow` (клампинг радиусов через `asymRadiiFromGroups`, наклон поверх), иначе старый
`buildHollowBox`. Прототип `scratchpad/asym-hollow-v2.js`: стресс 2000/2000 герметичны (5 «фейлов» —
`volume≤0` на «тонкая стенка + огромный радиус», меш при этом watertight; в HTML отсекает clamp).
Проверка в браузере: plain/outer/inner/taper/big-clamp — все watertight, рендер верный.

**Не сделано:** скруглённый кант борта (rolled lip — вертикальная фаска сходит в острый верх),
режим «борт»/лоток (`buildRimBox`) со скруглением.

## Этот проход: асимметричное скругление (12 радиусов) доведено до водонепроницаемости

Незавершённый эксперимент `asym-12-radius.js` (12 независимых радиусов рёбер) доработан до
герметичного меша на всех входных данных. Три точечные правки, все — в согласовании кусков,
не в математике (подробный разбор в `NOTES.md`):

1. `buildFaceAsymFromAxis` переписана: сетка по общим массивам осей + клип каждой ячейки по 4
   «margin»-функциям `u ≤ uHalf − min(ownR, distToCorner₊, distToCorner₋)`, совпадающим
   с касательной кромкой цилиндра (убрало щели/выступы и T-стыки старой сшивки).
2. `syncDiagonal`: исправлена константа диагонали (`vHalf − uHalf`, не зависит от радиуса —
   старая была неверна для углов со смешанными знаками) + итерация до неподвижной точки
   (три семейства осей теперь дозакрывают друг друга).
3. `buildEdgeCylinderFromAxis`: пропуск треугольников нулевой площади → у острого угла
   (`cornerMin = 0`) кромка сходится чистым веером вместо вырожденных квадов.

Проверка: `test-uniform-and-mixed.js` 16/16, `test-3radius-variant.js` 12/12, рандомный стресс
2000/2000 (радиусы до 30, resolution 5–16, 20 % нулевых рёбер) при допуске склейки 5 знаков.

**Интегрировано в HTML.** Функции портированы в `<script>` (vec-хелперы переиспользованы);
в UI куба — 3 группы «верх/низ/вертикали» (`asymRadiiFromGroups` → полный 12-радиусный R),
ветка `buildTrisForShape` выбирает asym при различии групп, иначе старый `genRoundedBox`; наклон
применяется поверх. Клампинг радиусов `K=0.98` (на пределе плоская грань вырождается). Проверка
в браузере (рендер + `manifoldCheck` в странице, все пути watertight) + near-limit стресс 4000/4000.

## Сделано

### Основа
Параметрический STL-генератор, самодостаточный HTML/CSS/JS. Куб — единственная фигура. Три взаимоисключающих режима дна/верха: сплошной, полый контейнер (`hollow`), борт/лоток (`rim`).
