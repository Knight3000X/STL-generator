# Parametric STL Generator

Самодостаточный HTML/CSS/JS инструмент для параметрического моделирования куба-контейнера
с экспортом в STL. Открывается прямо в браузере, без сборки и зависимостей — весь код
(включая Three.js для 3D-превью) лежит в одном файле `parametric-stl-generator.html`.

## Возможности

- **Пресеты**: панель кнопок над параметрами — «Сплошной / Скруглённый / Squircle / Надутый / Контейнер /
  Лоток / Бочка». Один клик задаёт скругления и режим дна/верха под типовую форму (снимает высокий порог
  входа при большом числе взаимоисключающих настроек скругления). Пресет сбрасывает только «оформительские»
  ключи (скругления, режим, стенка/борт) и применяет свой набор; размеры (X/Y/Z), сегменты, наклон и
  логотипы не трогает — форму можно доводить руками поверх пресета. Меши всех пресетов водонепроницаемы.

- **Применение по кнопке «Подтвердить»**: правки параметров и логотипов НЕ пересобирают модель
  на лету — они копятся, а к модели применяются только по нажатию кнопки «Подтвердить» (кнопка
  подсвечивается, пока есть неприменённые изменения). Это убирает тяжёлую перегенерацию на каждое
  нажатие/перетаскивание (был фриз на мобильном). Экспорт STL сначала применяет накопленные правки,
  чтобы файл соответствовал полям. Поворот (rx/ry/rz) остаётся живым (дешёвый путь без пересборки).
- **Несколько моделей (сборка)**: карточки моделей в панели — добавить (до 12), дублировать,
  удалить, скрыть/показать, переименовать. У каждой модели свои независимые параметры, логотипы,
  поворот, ПОЗИЦИЯ (смещение X/Y/Z, живое — без пересборки) и цвет превью. Клик по карточке делает
  модель активной — панели редактируют именно её (глобальное состояние сохраняется/загружается при
  переключении, `saveActiveModel`/`loadModel`; геометрия при переключении не пересобирается — берётся
  из кэша записи). Все ВИДИМЫЕ модели выгружаются в один STL со своими позициями/поворотами
  (`mergedExportTris`; несколько раздельных оболочек в одном STL — штатно для слайсеров). Модели не
  объединяются булево (CSG нет) — при пересечении печатаются как есть; это осознанное решение, а не
  недоделка: слайсеры корректно объединяют пересекающиеся оболочки при нарезке, а надёжный булев движок
  мешей в файле без зависимостей — крупный и хрупкий по герметичности алгоритм при малой практической
  пользе. Тест `test_multi_model.js`.
- **Сквозные отверстия** (вентиляция / кабель / винт): круглые вырезы насквозь по нормали выбранной грани.
  Только на ПЛОСКОМ сплошном кубе (без скругления, squircle, выгиба и логотипов — плоская стенка нужна для
  герметичности выреза; наклон допустим — отверстие наклоняется вместе со стенкой). Водонепроницаемо БЕЗ
  общего булева движка: каждая пробитая грань — это грид-квадраты + аннулюс (кольцо сетки грани ↔ окружность),
  а ствол переиспользует ту же выборку окружности, поэтому рёбра совпадают точно (`buildBoxWithHoles`,
  тест `test_holes.js`). Несколько отверстий на любые грани; сохраняются в JSON-конфиг.
- **Крышка для контейнера**: кнопка «+ Крышка для активной» создаёт модель-компаньон — неглубокий полый
  бокс, перевёрнутый открытой стороной вниз (`rx=180`), с внутренней полостью на зазор шире внешнего
  габарита активной модели (`+ стенка + печатный зазор ~0.4 мм/сторону), так что крышка надевается СВЕРХУ на
  контейнер. Переиспользует `buildHollowBox` → всегда водонепроницаема; ставится рядом в сборке, дальше
  настраивается как обычная модель.
- **Конфигурация сборки (JSON)**: кнопки «Сохранить (JSON)» / «Загрузить (JSON)» в блоке экспорта
  сериализуют ВСЮ сборку — все модели с параметрами, позициями, поворотами, цветами, видимостью и
  логотипами — в компактный JSON и загружают обратно. Логотипы хранят встроенное превью-изображение
  (PNG data URL), а heightmap при загрузке пересчитывается из него так же, как при загрузке картинки
  (файл остаётся небольшим). Импорт пересобирает сцену и геометрию всех моделей.
- **Три взаимоисключающих режима куба**: сплошной, полый контейнер (открытый верх, тонкие
  стенки), борт/лоток (сплошная основа + неглубокий поддон сверху).
- **Наклон стенок** (8 независимых углов) — работает во всех трёх режимах.
- **Скругление углов** на сплошном кубе — общий радиус ИЛИ **раздельно по 3 группам рёбер**
  (верх / низ / вертикали): если группы различаются, включается водонепроницаемый
  12-радиусный генератор (`buildAsymRoundedBox`, портирован из `asym-12-radius.js`; разбор и
  доказательство — в `NOTES.md`, тесты `test-uniform-and-mixed.js` / `test-3radius-variant.js`).
  Радиусы автоклампятся под размеры, наклон стенок совместим. Гладкость дуги задаёт «Сегменты скругления»
  (`filletSeg`), а плотность ОСЕВОЙ сетки (плоские участки между скруглёнными рёбрами) — отдельный параметр
  «Скругление: плотность осевой сетки» (`filletAxisRes`, 0 = авто = `2×filletSeg`); действует и на
  скруглённый полый/борт. **Логотипы совместимы со скруглением**
  на сплошном кубе (рельеф на плоских гранях; логотип клампится в плоскую зону, тест `test_fillet_logo.js`).
- **Squircle / супереллипс** (сплошной куб) — горизонтальное сечение становится супереллипсом:
  бока плавно выгибаются, а вертикальные углы скругляются в ЕДИНУЮ гладкую кривую (как иконка iOS),
  вытянутую по высоте с плоскими верхом/низом. Параметр «Squircle» 0..100 %: 0 — прямоугольник,
  ~45 — классический squircle, 100 — эллипс (`x=a·sign(cosθ)·|cosθ|^p`, `buildSquirclePrism`, тест
  `test_squircle.js`). Два ОТДЕЛЬНЫХ параметра «Squircle: скругление верха» и «Squircle: скругление низа»
  0..100 % дополнительно скругляют верхние/нижние рёбра НЕЗАВИСИМО — получается супереллипсоид («надутый
  куб», `buildSuperellipsoid` с раздельными экспонентами полюсов); 0 на стороне — плоская крышка там, так
  что верх и низ настраиваются по отдельности (например, куполообразный верх при плоском дне). Экватор
  (сечение на y=0) — одна и та же кривая при любых экспонентах, поэтому половинки сходятся без шва. Наклон
  совместим; отдельно от «скругления стенок» (то — подушка/горб на грани,
  это — единая кривая сечения). **Совместимо с логотипами**: рельеф embossed прямо на изогнутую стенку
  и оборачивается по кривизне (билдеры `buildSquirclePrismLogos`/`buildSuperellipsoidLogos`: поверхность
  строится сеткой, каждая вершина смещается вдоль локальной нормали; лого клампится от рёбер, так что
  швы сторона↔крышка/полюс остаются закрыты — водонепроницаемо, тест `test_squircle.js`). Верх/низ строятся
  БЕЗ полюсной/радиальной воронки (супереллипсоид — cube-sphere из 6 гранёв, плоские крышки — картезианская
  сетка `superellipseFillGrid`), поэтому логотип на верхушке/дне не размазывается в радиальные «спицы».
  **Сетка зонально сгущается**: грубая везде, мелкая только под самим логотипом (`zonedSamples` +
  `squircleABYZones`; у супереллипсоида — общие по осям cube-координаты, найденные грубым проходом), поэтому
  высокая детализация (до 500) стоит треугольников только под лого, а не по всей поверхности.
  **Работает во всех
  трёх режимах**: сплошной, полый контейнер (супереллипс-оболочка с тонкими стенками, открытый верх, плоский
  борт) и борт/лоток (супереллипс с неглубоким карманом сверху) — общий билдер `buildSquircleHollow`
  (внешняя стенка + полость `hw−t`/`hd−t` + дно + верхний борт-кольцо). Логотипы садятся на изогнутые внешние
  стенки, дно и пол полости/кармана. В полом/борте вертикальное скругление **«скругление низа» (squircleVBot)
  скругляет ДНО**
  контейнера (нижняя половина = нижняя половина супереллипсоида: прямые стенки сверху, куполообразное дно
  снизу; 0 — плоское дно, 100 — глубокая чаша). Верх остаётся открытым с плоским бортом (параметр
  «скругление верха» — только для сплошного куба). Полый — полость
  повторяет скруглённое дно (тонкая стенка), борт/лоток — только внешнее дно (карман сверху плоский). Дно
  строится кольцами по высоте для крутой части (ровный профиль) + **картезианская «шапка» на полюсе**
  (`superellipseDomeCapped`): центр полюса — обычная valence-4 вершина, поэтому **логотип на скруглённом дне
  (`-Y` / `-Y-inner`) тоже embossed** без полюсных «спиц». Перекрывает скругление углов и сетчатое дно.
- **Скругление стенок** — знаковый выгиб грани куполом: положительное значение выгибает наружу
  («бочка»), отрицательное прогибает внутрь («подушка»), 0 — плоско. Купол сходит в 0 к рёбрам грани,
  поэтому меш остаётся водонепроницаемым (тот же механизм смещения, что у логотипов). **Совместимо с
  логотипами** (рельеф логотипа складывается с выгибом грани), с наклоном И **со скруглением углов**
  (при заданных обоих грань выгибается куполом в плоской зоне между скруглёнными рёбрами — инсет-версия
  купола `makeInsetBulgeDisp`). Работает во всех режимах: сплошной куб (все 6 граней), полый контейнер (внешние
  стенки + дно, при этом **внутренняя полость повторяет выгиб — толщина стенки сохраняется**), борт/лоток
  (внешние стенки; лоток сплошной). Тест `test_wall_bulge.js`.
- **Логотипы/рельеф на гранях**: до 18 штук, независимый поворот, автоопределение
  инверсии по прозрачности/яркости изображения. Источник рельефа — **картинка ИЛИ текст**: поле «Текст
  рельефом…» рисует введённую строку в тот же heightmap, что и загруженная картинка (`makeTextLogoResult`:
  непрозрачные глифы на прозрачном фоне → рельеф берётся из альфа-канала), поэтому текст ведёт себя как
  обычный логотип — любая грань, поворот, все режимы/скругления, сохраняется в JSON-конфиг. С зональной
  сеткой (детализация только вокруг самого
  логотипа, не по всей грани) — на сплошном кубе, на скруглённом полом/борте И на ПЛОСКОМ полом/борте
  (грубая оболочка + мелкие подразбиения только под логотипом; верхний обод собран банд-сеткой, чтобы
  зональные точки в угловой полосе совпадали со стенками — тест `test_hollow_inner_logo.js`).
  **Совместимо со скруглением**
  во всех режимах: на сплошном кубе, на скруглённом полом контейнере и борте/лотке рельеф ложится
  на плоскую часть грани (внешние стенки, дно/пол полости, грань «−Y изнутри»), вписан между
  скруглёнными рёбрами (тесты `test_fillet_logo.js`, `test_fillet_logo_hollow.js`). **На squircle/
  супереллипсоиде** рельеф embossed прямо на изогнутую стенку и оборачивается по кривизне (лого выбирает
  грань по доминантной оси нормали, смещается вдоль локальной нормали). Наклон стенок совместим (при
  наклон+лого+скругление рельеф слегка тянется наклоном).
- **Сетчатое диагональное дно** полого контейнера (флажок «сетчатое ромбовидное дно») — садовая
  решётка: рёбра ±45°, ромбовидные сквозные отверстия, плоский профиль. Настраиваются шаг ромба,
  толщина ребра, сплошной кант. Водонепроницаемо (строится на общей сетке дна полости, воксельный узор
  с устранением защемлений, тест `test_lattice_floor.js`), совместимо с наклоном стенок и логотипами на
  стенках. Пока только на ПЛОСКОМ полом (не на скруглённом и не на борте).
- **Неблокирующая проверка герметичности**: индикатор «замкнутый ✓ / открытый ⚠» считается в **Web
  Worker** (off-thread), поэтому тяжёлая диагностика (последний источник «фриза» на главном потоке) больше
  не подвешивает UI. Источник воркера — сама `manifoldCheck`, сериализованная через `.toString()` (без
  сборки и дублирования); треугольники передаются transferable-буфером Float64 (бит-в-бит как inline).
  Если `Worker` недоступен — fallback на отложенную проверку. Потолок проверки поднят до 600k треугольников.
  (Саму генерацию геометрии оставили на главном потоке: в архитектуре «один файл без сборки» + Node-тесты,
  вызывающие геометрию напрямую, вынос генерации в воркер сломал бы либо то, либо другое — см. `PROGRESS.md`.)
- Экспорт в STL (Binary). ASCII убран как практически неиспользуемый (Binary компактнее, все слайсеры его едят).

## Структура

Всё лежит в корне репозитория (плоская раскладка, без подкаталогов):

```
parametric-stl-generator.html   — весь инструмент (открыть в браузере)
run-all.sh                      — запускает весь набор тестов одной командой
PROGRESS.md                     — журнал разработки: что сделано и почему, по каждому проходу
NOTES.md                        — разбор 12-радиусного скругления и доказательство герметичности
asym-12-radius.js               — автономный порт 12-радиусного билдера (источник in-page версии)
stub_preamble.js                — DOM/rAF-стабы, чтобы код страницы запускался под Node
test_*.js, test-*.js            — Node-тесты геометрии и управляющего потока (см. ниже)
minimal-repro-missing-breakpoint.js — минимальный repro одного бага сшивки грани (для справки)
```

## Тесты

Вся геометрия (углы куба, швы между гранями, водонепроницаемость меша) проверяется в
Node.js перед тем, как попасть в файл — это соглашение действует для каждой фичи в этом
проекте, не только оформлено постфактум.

```
./run-all.sh
```

запускает все наборы тестов (**513 проверок в 18 файлах**) и выводит per-file и общий
итог (ненулевой код возврата при любом падении). Большинству тестов нужен реальный код
из `parametric-stl-generator.html` (не переписанная копия) — `run-all.sh` сам вытаскивает
`<script>`, подставляет DOM-стабы (`stub_preamble.js`), а для `test_debounce_flow.js` и
`test_apply_button.js` ещё и подменяет `regenerate()`/`applyRotationOnly()` на счётчики. Ручная
процедура для каждого файла описана в его собственном заголовке, если нужно запустить один тест отдельно.

`test_logo_zone.js` — единственный самодостаточный тест (не требует извлечения кода из
HTML, содержит собственные копии проверяемых функций): `node test_logo_zone.js`.

## Известные ограничения

- **Полый контейнер И режим «борт»/лоток** полностью скругляются: внешние (низ+вертикали+верхний
  кант `filletTop`) и внутренние полости/кармана (дно+вертикали+кант `filletInnerLip`) рёбра —
  водонепроницаемо. Трей = тот же билдер с мелким карманом (`pocketDepth=rimHeight`). Скругление
  **совместимо с логотипами** (рельеф на внешних стенках + дне полости/кармана, вписан в плоскую зону;
  тест `test_fillet_logo_hollow.js`).
- **Верхний кант**: `filletInnerLip>0` — весь верх закатан (скруглённый внешний `filletTop` +
  скруглённый внутренний кант); `filletInnerLip=0` — весь верх **острый плоский**: обе оболочки со
  скруглёнными вертикальными углами, но острым плоским ободом, без конусов-«шпилей» и без выступающих
  «рожков» на внешних углах (билдер `buildSharpRimHollow`: обе оболочки поднять на радиус вертикали и
  срезать по плоскости проёма; борт = явное кольцо между граничными петлями оболочек). Клампы для
  валидности: внутренняя вертикаль ≤ глубина полости, внешняя вертикаль ≤ внутр.+стенка. Любой суб-0.1
  мм радиус снапится в острое ребро. Пути покрыты `test_rounded_fillet.js`; Node-стресс острого
  билдера — 2000/2000 водонепроницаемы и ориентационно-согласованы.
- Зональная сетка для логотипов теперь есть во ВСЕХ режимах (сплошной, скруглённый и ПЛОСКИЙ полый/борт):
  оболочка грубая, мелкая детализация — только под логотипом, поэтому высокую детализацию логотипа можно
  крутить без перегенерации всей оболочки. Плотность зон во всех режимах задаёт ОДНА общая настройка
  «Детализация логотипов» (`logoResolution`). Плоский полый/борт без логотипов остаётся на равномерной сетке.
- **Сетчатое диагональное дно** — только для ПЛОСКОГО полого контейнера, и это осознанное решение, а не
  недоделка: решётка со СКВОЗНЫМИ отверстиями осмысленна лишь там, где под дном пустота. У борта/лотка под
  карманом сплошная база (сквозному отверстию некуда вести), у скруглённого полого дно — купол (ровная
  диагональная решётка по кривизне определена плохо). **Профиль ребра**: плоский ИЛИ **скруглённый**
  (флажок «Сетка: скруглённый профиль ребра») — верх ребра куполится к отверстиям. Скругление сделано
  как поле высот по ВЕРШИНАМ сетки (дистанция до ближайшей дыры, BFS): соседние ячейки читают общую высоту
  на общем ребре, поэтому шва нет — герметично (пер-ячейковые инсеты рвали бы швы; проверено `test_lattice_floor.js`).
  **Логотипы совместимы с решёткой**: на стенках — как обычно; логотип на дне полости («−Y изнутри»)
  тоже рисуется — сплошной становится только САМА ФОРМА рельефа (там, где `floorFn≠0`), а фон остаётся
  сеткой, поэтому картинка «висит» на фоне решётки, а не на сплошном квадрате-плитке. Форс-solid идёт ДО
  устранения защемлений (любые «шахматки» от тонкой формы вычищаются), высота верха эмбоссится ПО ВЕРШИНАМ
  → герметично, работает и со скруглённым профилем ребра (тест `test_lattice_floor.js`). Логотип на внешнем
  дне («−Y») с решёткой пока не поддержан. Шаг ромба (`cell`) и толщина ребра заданы в мм и
  расцеплены с детализацией стенок (= «Детализация логотипов»): смена размера ромба не огрубляет
  стенки. Диагональ имеет мелкую «лесенку» по краям — чем выше детализация, тем глаже.
- Толщина стенки при наклоне не идеально постоянна по нормали (расхождение небольшое для
  разумных углов, см. `PROGRESS.md`).
- Экстремальный наклон + логотип у самого края сплошного куба может дать негерметичный шов.

Подробная история и обоснование инженерных решений — в `PROGRESS.md`.
